## 一、架构解耦

### 问题A1：main.py 是上帝文件（884行，承载6种职责）

【现状问题】
`main.py`（884行）同时承担：HTTP路由定义、SSE流式响应编排、LLM调用线程管理、图片生成编排、文件归档逻辑、报告生成调度。这6种职责分属3个不同的业务域（API层、编排层、持久化层），全部压缩在单一文件中。

【违反标准】
- 违反 ARCH-001（业务功能独立拆分为独立模块，禁止写入主架构文件）
- 违反 ARCH-004（单模块内部只存放同一业务域逻辑）
- 因果链：[背景]：主架构文件承担超过3种职责后，任何一个职责的修改都需要通读整个文件才能理解影响范围 → [为什么构成问题]：当前 884 行中，SSE流程控制逻辑与业务逻辑深度交织，修改图片生成流程（行449-482）可能意外影响对话保存逻辑（行297-324），因为两者共享同一个 `generate()` 协程作用域 → [不改的长期风险]：每新增一个 Node 节点，main.py 预计增长100-150行，10个Node时文件将超过2000行，单文件内的变量作用域冲突概率指数上升

【修正方案】
1. 将 `generate()` 协程中的SSE流式控制逻辑抽离为 `server/orchestrator/sse_engine.py`，仅暴露 `async def stream_response(node_id, message, history)` 接口
2. 将图片生成逻辑（行449-482）抽离为 `server/nodes/image_generator.py`
3. 将文件归档逻辑（行297-424）抽离为 `server/orchestrator/archiver.py`
4. 修改的底层原因：SSE流控制是横切关注点，与业务Node的具体行为无关。解耦后，新增Node只需注册到SSE引擎，不需要修改main.py，符合开闭原则

### 问题A2：common.py 收纳5种无关职责（515行）

【现状问题】
`server/utils/common.py` 同时包含：Markdown→HTML转换（行7-94）、HTML文件生成（行111-147）、文件系统I/O（行150-158）、数据库CRUD（行185-293）、SVG Logo生成（行301-353）、Logo prompt提取（行356-379）。这5种职责对应5个完全不同的关注点。

【违反标准】
- 违反 ARCH-004（单模块内部只存放同一业务域逻辑）
- 因果链：[背景]：utils/common 是最容易退化为"什么都往里扔"的模块 → [为什么构成问题]：Markdown渲染（纯函数）和数据库操作（有状态、有连接管理）共享同一文件，引入任何依赖（如pymysql）就污染了纯函数模块 → [不改的长期风险]：utils/common 持续膨胀，最终任何新功能开发者都会"先放common里"，失去模块拆分的所有收益

【修正方案】
1. 拆分为：`server/utils/markdown.py`（markdown_to_html, _escape_html, _render_inline）
2. 拆分为：`server/utils/file_io.py`（generate_html_file, save_project_file, save_project_file_bytes）
3. 拆分为：`server/utils/svg.py`（_generate_logo_svg, _extract_logo_prompts）
4. 拆分为：`server/persistence/database.py`（get_db, save_conversation等全部DB操作）
5. 修改的底层原因：纯函数工具、文件I/O、数据库持久化是三个不同的变更频率和测试边界。纯函数应无副作用可单元测试；数据库操作需要集成测试环境。混在一起时，改一行Markdown渲染逻辑就需要担心数据库连接池是否被意外初始化

### 问题A3：reports.py 在函数体内动态导入，形成隐藏循环依赖风险

【现状问题】
`server/reports.py` 第85行在 `generate_summary_report()` 函数体内执行：
```python
from server.intent.recognizer import get_uploaded_files_text
```

【违反标准】
- 违反 ARCH-006（模块之间无强双向依赖）
- 违反 ARCH-007（跨模块调用统一走通用中间层）
- 因果链：[背景]：函数体内的延迟import通常是为了打破循环依赖，但它掩盖了真正的架构问题 → [为什么构成问题]：reports.py 依赖 intent/recognizer.py 的内部工具函数，而 recognizer.py 本身是一个意图识别模块，不应该被报告生成模块直接依赖。两个模块之间没有合理的业务域关联 → [不改的长期风险]：一旦 reorganize intent 模块结构，rename 了 `get_uploaded_files_text`，reports.py 会在运行时（而非导入时）才报错，且报错时用户正在等待报告生成，体验极差

【修正方案】
1. 将 `_uploaded_files` 从 intent/recognizer 移到独立的 `server/state/session.py`
2. reports.py 通过依赖注入接收文件上下文，而非直接 import 内部函数
3. 修改的底层原因：文件上传状态是跨模块共享的数据，不是意图识别模块的私有数据，它应该属于统一的会话状态层

### 问题A4：前端 App.tsx 是上帝组件（1687行，56个useState）

【现状问题】
`kaiwu/src/App.tsx` 共1687行，承载：路由管理、对话流SSE处理、会话缓存、状态管理（56个useState）、文件上传、技能库、项目库、设置页、AI生图、AI视频、AI编程——全部在一个组件内。

【违反标准】
- 违反 ARCH-001（业务功能独立拆分为独立模块）
- 因果链：[背景]：React单组件承载所有页面逻辑是最常见的初期架构反模式 → [为什么构成问题]：56个useState导致重渲染范围不可控——修改 `imageCount` 状态会触发整个1687行组件树的重渲染，包括与图片完全无关的对话区、技能库、设置页 → [不改的长期风险]：新增一个页面需要在这个1687行文件中找到正确位置插入状态+UI+逻辑，改错一个括号会导致整个App崩溃

【修正方案】
1. 按页面路由拆分为独立Page组件：`pages/HomePage.tsx`, `pages/SkillsPage.tsx`, `pages/ProjectsPage.tsx`, `pages/SettingsPage.tsx`, `pages/CodingPage.tsx`
2. 共享状态（当前对话、用户信息）通过React Context或状态管理库注入
3. 修改的底层原因：每个Page组件独立管理自己的状态，互不干扰。React的reconciliation可以在Page级别切断不必要的重渲染

### 问题A5：LLM客户端路由硬编码，无法扩展

【现状问题】
`server/llm_client/router.py` 第8行：
```python
if model and "doubao" in model.lower():
```
LLM提供商的选择逻辑通过字符串匹配硬编码，新增一个提供商需要修改router.py。

【违反标准】
- 违反 ARCH-002（模块仅通过统一抽象接口与主架构通信）
- 因果链：[背景]：硬编码的if/else路由在提供商数量=2时够用 → [为什么构成问题]：新增第三个提供商（如通义千问）时，必须修改router.py的核心分支逻辑，且需要新增API_KEY导入 → [不改的长期风险]：router.py逐步膨胀为"所有LLM提供商的配置+调用逻辑"，违反开闭原则，也无法支持运行时动态切换

【修正方案】
1. 定义 LLMProvider 抽象接口
2. DeepSeek和Doubao各实现此接口
3. 使用注册表模式：`PROVIDER_REGISTRY = {"deepseek": DeepSeekProvider(), "doubao": DoubaoProvider()}`
4. router通过注册表查找，不硬编码if/else
5. 修改的底层原因：注册表模式让新增提供商只需注册一个实现类，不需要修改任何已有代码。符合开闭原则

## 二、Spec约束

### 问题S1：零Spec基线——项目无任何冻结决策文档

【现状问题】
项目目前没有任何Spec文档。所有架构决策（Node拆分维度、SSE事件协议、intent路由规则）都仅存在于代码中，无任何显式记录。

【违反标准】
- 违反 SPEC-001（决策需以Spec形式冻结）
- 违反 SPEC-002（确定落地Spec四要素必须完整）
- 因果链：[背景]：项目处于早期快速迭代阶段，口头+代码=够快 → [为什么构成问题]：当前代码中的Node拆分（node1/1.5/2/3/3.1/4/5）和依赖关系（DEPENDENCIES字典）没有任何文档解释"为什么node1.5排在node2之后"、"为什么node4依赖node1+node2但不强制依赖node3" → [不改的长期风险]：新人（或6个月后的自己）不理解依赖链的设计意图，可能错误地"优化"掉关键依赖，导致营销方案生成时缺少市场数据

【修正方案】
1. 为现有6个关键决策创建冻结Spec：
   - Spec-001: Node拆分与依赖链设计
   - Spec-002: SSE事件协议与前端约定
   - Spec-003: Intent识别四层匹配策略
   - Spec-004: 会话状态管理与缓存策略
   - Spec-005: 文件归档双路径规则
   - Spec-006: API密钥管理（绝对不做：硬编码）
2. 每份Spec按 `docs/02-Spec模板.md` 格式填写
3. 修改的底层原因：这些决策每天被代码引用，却没有任何正式记录。一旦需要做架构调整（如调整Node依赖链），没有基线文档可以对照评估改动影响

## 三、决策一致性

### 问题D1：DATA_INTEGRITY 常量定义但未使用

【现状问题】
`server/config.py` 第57-61行定义了 `DATA_INTEGRITY` 常量，包含数据真实性铁律的文本。`main.py` 第30行将其导入，但在整个代码中从未引用过 `DATA_INTEGRITY`。

【不一致分析】
- 设计意图：config.py中定义DATA_INTEGRITY表明设计者希望有一个全局的数据真实性约束
- 实际实现：prompts.py中的每个Node system prompt各自内嵌了数据真实性规则，但没有引用全局DATA_INTEGRITY
- 偏离原因：DATA_INTEGRITY在config.py定义后被遗忘，prompts迭代时各自手写了数据规则
- 因果链：[当初决策的因果]：建立统一的数据真实性铁律可以保证所有Node遵守同一标准 → [这个偏离如何破坏了设计初衷]：现在修改数据真实性规则需要同时改DATA_INTEGRITY常量+6个Node prompt，几乎可以确定不会有人同时改两处，规则将逐渐分化

【修正方案】
1. 在 `generate_ai_response()` 中将 `DATA_INTEGRITY` 拼接到 system prompt 前缀，全局生效
2. 从各Node prompt中删除重复的数据规则文本
3. 修正的底层原因：单一真相源原则——数据真实性规则只有一处定义，修改时改一处影响全局

## 四、禁止功能

### 问题F1：API密钥硬编码在源代码中

【现状问题】
`server/config.py` 第7行和第14行：
```python
DEEPSEEK_API_KEY = "sk-xxxx"
DOUBAO_API_KEY = "ark-xxxx"
```

【违规分析】
- 虽然没有对应的"绝对不做"Spec（因为还无Spec体系），但这是业界公认的安全红线
- 硬编码的API密钥已经暴露在源代码中，如果代码被提交到任何Git仓库（即使是private），密钥就永久存在于Git历史中
- 因果链：[背景]：快速开发阶段为方便把密钥写在代码里 → [问题]：密钥与代码绑定，无法按环境区分（开发/测试/生产），无法独立轮换密钥 → [如果放任]：密钥泄露后必须同时修改代码并重新部署才能轮换，且Git历史中的密钥无法真正删除

【修正方案】
1. 立即创建 Spec-006（绝对不做类型）：禁止API密钥硬编码
2. 将所有API密钥移至环境变量或 `.env` 文件（`.env` 加入 `.gitignore`）
3. 修改config.py为读取环境变量，提供默认值仅用于本地开发
4. 修正的底层原因：密钥与代码分离是安全基线。轮换密钥不应要求代码变更
