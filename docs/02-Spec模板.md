# 项目决策 Spec 模板（标准化）

> **冻结状态**：已冻结定稿 | **用途**：所有项目决策的唯一记录格式 | **最后更新**：2026-06-30

---

## 使用说明

**为什么需要标准化模板**：项目决策分散在口头沟通、聊天记录、代码注释中，无法作为开发/Review的权威依据。标准化模板强制决策写全因果链，Review 时只需比对 Spec → 代码，无需重构决策过程。

**缺少模板的协作问题**：每个决策格式不同，关键字段缺失（最常见：没有"功能边界"），导致不同人对同一个 Spec 的理解不同，Review 无统一标准。

---

## 类型 1：确定要落地的 Spec

```markdown
# Spec-[编号]: [一句话描述决策内容]

| 字段 | 内容 |
|------|------|
| Spec 类型 | 确定落地 |
| 冻结状态 | 已冻结 |
| 创建日期 | YYYY-MM-DD |
| 关联模块 | [模块名称] |
| 关联 Spec | [依赖或前置Spec编号，如无可写"无"] |

## 一、背景与现存问题

[描述决策产生的上下文：当前项目处于什么阶段，遇到了什么问题触发这个决策]

## 二、最终实现方案

[具体怎么实现，足够清晰，新成员无需额外沟通即可理解]

## 三、做出该选择的底层原因（因果链）

1. 业务原因：[这个选择对业务目标、用户体验、交付节奏的影响]
2. 技术原因：[技术可行性、性能、可维护性的考量]
3. 对比过的备选方案：
   - 备选A：[简述] → 弃用原因：[…]
   - 备选B：[简述] → 弃用原因：[…]

## 四、落地硬性约束

- [约束1：例如"必须使用 XXX 接口通信"]
- [约束2：例如"禁止绕过 XX 中间层"]
- [约束3：例如"单文件不超过 XXX 行"]

## 五、功能边界

### 覆盖场景
- [场景1]
- [场景2]

### 明确不覆盖
- [不覆盖1]
- [不覆盖2]

## 六、不这么做会带来的风险

1. [风险1：具体后果 + 影响范围]
2. [风险2：具体后果 + 影响范围]
```

## 类型 2：明确绝对不做的 Spec

```markdown
# Spec-[编号]: [禁止的行为/方案描述]

| 字段 | 内容 |
|------|------|
| Spec 类型 | 绝对不做 |
| 冻结状态 | 已冻结 |
| 创建日期 | YYYY-MM-DD |
| 关联模块 | [受此禁令约束的模块] |
| 关联 Spec | [相关Spec编号] |

## 一、禁止的功能 / 技术方案

[清晰描述：什么绝对不能做，什么技术方案绝对不能选]

## 二、否决的完整原因（因果链）

1. 触发事件：[什么具体事件/讨论触发了这个否决]
2. 否决原因：
   - 业务层面：[…]
   - 技术层面：[…]
   - 架构层面：[…]
3. 曾考虑的相关方案：[简述为何即便修改后仍不可行]

## 三、长期约束规则

- [约束1：持续有效的禁令边界]
- [约束2：不得以什么方式绕过]

## 四、变更该决策需要的评审条件

满足以下**全部条件**方可重新评审此 Spec：
- [ ] [条件1：例如"业务假设发生实质变化"]
- [ ] [条件2：例如"有充分数据证明原否决原因已不存在"]
- [ ] [条件3：例如"项目架构 Review 会议通过"]

## 五、违反此 Spec 造成的后果

1. [后果1]
2. [后果2]
```
