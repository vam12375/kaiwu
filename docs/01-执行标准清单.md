# 曜势科技（Kaiwu）项目执行标准清单

> **冻结状态**：已冻结定稿 | **适用范围**：全部开发/Review/文档阶段 | **最后更新**：2026-06-30

---

## 一、架构规范

### 1.1 模块解耦强制规则

| 编号 | 标准 | 因果链 |
|------|------|--------|
| ARCH-001 | 业务功能独立拆分为独立模块，禁止写入主架构文件 | **背景**：主架构文件（App.tsx / main.py）随功能增长会膨胀为"上帝文件"，单文件超2000行后定位Bug时间指数级上升。**选择理由**：独立模块天然具备边界，修改一个业务域不影响其他。**不这么做的风险**：主架构文件耦合所有业务逻辑后，改一行代码可能触发全局回归，无法并行开发 |
| ARCH-002 | 模块仅通过统一抽象接口与主架构通信 | **背景**：直接操作全局变量造成隐式依赖，新增功能时需要全局搜索变量引用。**选择理由**：接口是唯一契约，模块内部可以自由重构。**不这么做的风险**：全局变量被多模块同时修改，出现时序依赖Bug，且无法定位责任模块 |
| ARCH-003 | 禁止模块直接操作主架构全局变量、底层核心对象 | **背景**：全局状态被散布在各模块中修改，Review时无法追踪数据流向。**选择理由**：数据流单向可控，所有状态变更可审计。**不这么做的风险**：状态污染链式扩散，一个模块的Bug通过全局变量传染给所有模块 |

### 1.2 高内聚规则

| 编号 | 标准 | 因果链 |
|------|------|--------|
| ARCH-004 | 单模块内部只存放同一业务域逻辑 | **背景**：业务域混杂的模块无法独立测试、独立部署、独立交付出。**选择理由**：按业务域拆分后，每个模块有清晰Owner，问题直接定位。**不这么做的风险**：一个"common/utils"模块收纳所有零散逻辑，最终变成第二个主架构，失去拆分意义 |
| ARCH-005 | 模块内部允许子目录分层（handler/service/repository），但禁止跨业务域 | **背景**：一个业务域内部也需要分层管理职责。**选择理由**：分层是模块内部的私有实现，不影响外部耦合。**不这么做的风险**：过度扁平化导致单文件过大；过度嵌套导致无效抽象 |

### 1.3 低耦合规则

| 编号 | 标准 | 因果链 |
|------|------|--------|
| ARCH-006 | 模块之间无强双向依赖 | **背景**：A依赖B、B同时依赖A形成循环引用，JavaScript/Python都可能运行时炸掉。**选择理由**：单向依赖保证拓扑顺序，可以单独编译/加载。**不这么做的风险**：循环依赖导致模块加载顺序不确定，生产环境间歇性报错，难以复现 |
| ARCH-007 | 跨模块调用统一走通用中间层（事件总线 / 消息队列 / 依赖注入容器） | **背景**：模块间点对点调用形成网状依赖，新增模块需要改动N个已有模块。**选择理由**：中间层是星型拓扑，新增模块只连中间层。**不这么做的风险**：N个模块的M×N依赖关系，模块数到20个以上时架构不可维护 |
| ARCH-008 | 每次拆分输出必须说明：设计收益 + 不解耦的长期风险 | **背景**：无因果的拆分会被后续开发者误解为过度工程化而合并回去。**选择理由**：显式因果保证架构决策可传承。**不这么做的风险**：6个月后新成员不理解拆分原因，逐步合并回单体，架构退化 |

## 二、文档索引规范

| 编号 | 标准 | 因果链 |
|------|------|--------|
| DOC-001 | 文档索引覆盖六类：架构文档、决策Spec、模块设计、接口文档、测试文档、Review记录 | **背景**：缺少索引时，新成员只能通过文件名猜测文档内容。**选择理由**：索引是一张地图，告诉每个人什么文档在什么地方、什么状态。**不这么做的风险**：重复创建同类文档；已有决策被新文档覆盖；Spec和实现脱节 |
| DOC-002 | 每条索引包含：文件名、存储路径、用途、关联模块、冻结状态 | **背景**：仅文件名无法判断文档是否可信。**选择理由**：五个字段最小化满足检索+可信度判断需求。**不这么做的风险**：引用草稿文档做决策依据；不知道文档覆盖范围产生遗漏 |
| DOC-003 | 明确区分「已冻结定稿」与「草稿迭代」两类状态 | **背景**：Spec冻结后修改需要评审流程，草稿可以自由迭代，混淆两种状态导致流程混乱。**选择理由**：冻结=只读+Review依据；草稿=可写入+不可作为Review依据。**不这么做的风险**：用草稿Spec去Review代码，Review结论无效 |
| DOC-004 | 每类文档写明：为什么需要独立建立 + 缺少会带来什么协作问题 | **背景**：团队成员不理解文档价值时会跳过不写。**选择理由**：显式因果降低文档维护阻力。**不这么做的风险**：文档流于形式，关键决策存在于聊天记录中，无法追溯 |
| DOC-005 | 新增任何Spec、模块文档，自动同步更新索引 | **背景**：索引过时比没有索引更危险（索引指向错误的冻结状态）。**选择理由**：维护索引是创建文档的最后一步。**不这么做的风险**：过时索引引导做出错误决策 |

## 三、Spec编写规范

| 编号 | 标准 | 因果链 |
|------|------|--------|
| SPEC-001 | Spec分两大类：确定要落地的 / 明确绝对不做的 | **背景**：所有决策都需要双向记录——做什么和不做什么同样重要。**选择理由**："不做什么"防止未来有人重新提案已否决的方案。**不这么做的风险**：已否决方案被新人重新提出，重复浪费评审时间 |
| SPEC-002 | 确定落地Spec四要素：最终方案、底层原因、硬性约束、功能边界 | **背景**：只写方案不写原因，后来者不理解为什么选A不选B，实施时偏离初衷。**选择理由**：四要素构成完整决策闭环——做什么、为什么、怎么约束、边界在哪。**不这么做的风险**：功能边界模糊导致范围蔓延，一个"用户登录"最终变成"用户中心全功能" |
| SPEC-003 | 绝对不做Spec四要素：禁止项、否决原因、长期约束、变更评审条件 | **背景**：否决记录缺失导致被否决方案在新场景下被重新实施。**选择理由**："绝对不做"是一个承诺，变更条件是一个安全阀。**不这么做的风险**：业务场景变化时无法判断是否可以重新评估，要么死守过时禁令，要么随意突破 |
| SPEC-004 | 所有Spec必须配套因果链：背景→现存问题→选择理由→不做的风险 | **背景**：无因果的Spec等于"领导说了算"，无法参与理性讨论。**选择理由**：因果链让Spec可辩论、可演进。**不这么做的风险**：Spec变成教条，当假设条件变化时无人敢改 |
| SPEC-005 | 所有Spec冻结后，开发/Review以此为唯一依据，禁止擅自扩缩边界 | **背景**：Spec和实现脱节是项目最常见的质量漏洞来源。**选择理由**：冻结Spec=唯一真相源，代码要么对齐Spec，要么走流程更新Spec。**不这么做的风险**：代码实现超出Spec范围的行为，无人知晓，Review无法检测 |

## 四、AI Review规范

| 编号 | 标准 | 因果链 |
|------|------|--------|
| REVIEW-001 | 每完成一个功能/模块，必须对照冻结Spec执行Review | **背景**：功能完成后无人校验，Spec与实现逐步偏离。**选择理由**：及时Review的成本远低于后期重构。**不这么做的风险**：3个月后Spec文档和实际代码已经完全不同，文档作废 |
| REVIEW-002 | 校验四维度：模块解耦、Spec约束遵守、逻辑与决策初衷一致、无私自新增禁止功能 | **背景**：单一维度Review（如只检查语法）无法发现架构层面的偏离。**选择理由**：四维度覆盖从架构到细节的完整质量保证。**不这么做的风险**：代码符合语法但违反架构约束，短期无问题但长期积累技术债 |
| REVIEW-003 | Review结论固定格式：现状问题→违反的Spec+对应因果→修正方案+修正原因 | **背景**：只说"这里不对"不给原因的Review无法执行整改。**选择理由**：格式强制Review从"发现"走到"解决方案"。**不这么做的风险**：Review流于形式，"知道了"但不会修改 |
| REVIEW-004 | Review结论必须沉入项目文档索引，归档为Review记录 | **背景**：Review结果散落在聊天记录中无法回溯。**选择理由**：归档Review形成项目质量演进轨迹。**不这么做的风险**：同样的问题被反复Review，每次都要重新解释 |
