# AI Review 校验模板

> **冻结状态**：已冻结定稿 | **用途**：每个功能/模块完成后的强制执行Review流程 | **最后更新**：2026-06-30

---

## 为什么需要独立的Review模板

代码Review通常聚焦于语法和逻辑错误，但本项目需要Review四个更深层维度：架构解耦、Spec一致性、决策初衷对齐、禁止功能检测。这四个维度语法检查无法覆盖，必须有结构化Review流程。

**缺少此模板的协作问题**：Review变成走过场——"看了一下没问题"——实际上架构违规和Spec偏离没有被检测。

---

## Review 执行时机

| 触发条件 | Review类型 | 频率 |
|----------|-----------|------|
| 一个业务模块开发完成 | 完整四维度Review | 每模块必执行 |
| 修改已有模块 | 增量Review（仅变更涉及的维度） | 每次PR |
| 新增Spec文档 | Spec完整性Review | 新增即执行 |
| 合并到主分支前 | 汇总Review | 每次合并 |

## 校验维度 1：架构解耦检查

### 检查项

| 编号 | 检查内容 | 为什么需要这项检查（因果说明） |
|------|----------|------------------------------|
| A1 | 新增代码是否全部在独立模块目录内，没有写入主架构文件？ | **背景**：主架构文件膨胀导致单文件难以维护。**不检查的风险**：开发者为图省事直接在 App.tsx 里加逻辑，3次迭代后回到单体结构 |
| A2 | 模块是否通过统一抽象接口与主架构通信，而非直接引用全局变量？ | **背景**：全局变量隐式依赖是最隐蔽的耦合形式。**不检查的风险**：看似"能用"，但全局变量被其他模块意外修改时，Bug无法归因 |
| A3 | 模块内部是否单一业务域？有无混入其他域的逻辑？ | **背景**：跨域模块无法独立测试。**不检查的风险**：模块A的测试用例需要先启动模块B，测试变成集成测试，失去快速反馈 |
| A4 | 模块间是否存在双向依赖？ | **背景**：双向依赖导致循环引用。**不检查的风险**：模块加载顺序不确定，生产环境偶发 undefined 错误 |
| A5 | 跨模块调用是否走通用中间层？ | **背景**：点对点调用形成网状依赖。**不检查的风险**：移除一个模块需要同时修改N个依赖它的模块 |

### 输出格式

```
【架构解耦 - 现状问题】
1. [具体问题：哪个文件、哪个位置、违反了什么]

【违反标准】
- 违反 ARCH-00X [标准名称]
- 因果链：[背景] → [为什么这构成问题] → [不改的长期风险]

【修正方案】
1. [具体修改步骤]
2. [修改的底层原因：为什么这个方案是正确选择]
```

## 校验维度 2：Spec 约束遵守检查

### 检查项

| 编号 | 检查内容 | 为什么需要这项检查（因果说明） |
|------|----------|------------------------------|
| S1 | 实现是否覆盖了 Spec "覆盖场景"中的全部场景？ | **背景**：Spec 定义覆盖场景但代码未实现是遗漏Bug。**不检查的风险**：Release时发现核心场景未实现 |
| S2 | 实现是否没有超出 Spec "明确不覆盖"的边界？ | **背景**：超出边界的"额外功能"未经过设计评审。**不检查的风险**：看似贴心的额外功能引入了新的安全漏洞或性能问题 |
| S3 | 实现是否遵守 Spec 中的全部硬性约束？ | **背景**：硬性约束是架构安全底线。**不检查的风险**：绕过约束的代码暂时能跑，但破坏了架构的完整性假设 |
| S4 | 如果Spec类型是"绝对不做"——代码中是否完全不存在被禁止的功能？ | **背景**：被否决的方案可能在重构中无意被重新引入。**不检查的风险**：被否决的旧方案借新功能之名复活 |

### 输出格式

```
【Spec约束 - 现状问题】
1. [具体问题：哪个实现违反了哪条Spec]

【违反Spec】
- 违反 Spec-[编号] 第[X]条
- Spec原文约束：[引用Spec原文]
- 因果链：[Spec当初为什么做出这个约束] → [违反会触发什么问题]

【修正方案】
1. [具体修改步骤]
2. [修正的底层原因：为什么这个实现必须回归Spec边界]
```

## 校验维度 3：决策初衷一致性检查

### 检查项

| 编号 | 检查内容 | 为什么需要这项检查（因果说明） |
|------|----------|------------------------------|
| D1 | 代码的逻辑走向是否符合 Spec 中"做出该选择的底层原因"？ | **背景**：实现细节可能与高层设计意图偏离。**不检查的风险**：代码"能跑"但方向不对——比如Spec设计的是事件驱动架构，代码实现成了轮询 |
| D2 | 关键数据结构/接口设计是否反映了Spec中的设计意图？ | **背景**：接口是最容易"走样"的设计元素。**不检查的风险**：接口设计偏离会导致调用方不得不适配不合理的API，形成连锁污染 |
| D3 | 是否有"我个人觉得这样更好"的私自改动，但未更新Spec？ | **背景**：Developer在实现阶段的私自决策如果不经 Review 直接固化为代码，就绕过了Spec流程。**不检查的风险**：Spec形同虚设，决策权威从Spec转移到个别开发者的临时判断 |

### 输出格式

```
【决策一致性 - 现状问题】
1. [具体不一致点]

【不一致分析】
- Spec链接：[Spec编号] 中的设计决策原文
- 实际实现：[代码做了什么]
- 偏离原因：[为什么会出现这个偏离]
- 因果链：[Spec当初决策的因果] → [这个偏离如何破坏了设计初衷]

【修正方案】
1. [方案]
2. [修正原因]
```

## 校验维度 4：禁止功能检测

### 检查项

| 编号 | 检查内容 | 为什么需要这项检查（因果说明） |
|------|----------|------------------------------|
| F1 | 代码中是否存在"绝对不做" Spec 中列出的功能/技术方案？ | **背景**：被否决的方案可能被以不同形式重新实现。**不检查的风险**：历史的坑被重新挖开 |
| F2 | 是否有超出模块边界的"顺手做的"功能？ | **背景**：未规划的功能缺少测试和文档。**不检查的风险**：隐藏功能在生产环境被意外触发 |
| F3 | 是否有绕过中间层的"快捷调用"？ | **背景**：开发者可能会绕过中间层直接调用来省时间。**不检查的风险**：捷径破坏架构约束，形成单点依赖 |

### 输出格式

```
【禁止功能 - 现状问题】
1. [检测到的违规项]

【违规分析】
- 违反 Spec-[编号]：[禁止的具体行为]
- 因果链：[当时否决的原因] → [如果放任此违规会导致什么]

【修正方案】
1. [方案]
2. [修正原因]
```

## Review 记录归档规则

1. 每次Review结论保存为 `docs/reviews/YYYY-MM-DD-模块名-review.md`
2. 同步更新 `docs/03-项目文档索引.md` 的"功能Review记录"表格
3. 若Review结果为 ❌ 不通过：禁止合并代码，整改完成后重新Review
4. 若Review结果为 ⚠️ 条件通过：P0问题已修复，P1/P2创建跟踪Issue
