# Spec-001: Node 拆分与依赖链设计

| 字段 | 内容 |
|------|------|
| Spec 类型 | 确定落地 |
| 冻结状态 | 已冻结 |
| 创建日期 | 2026-06-30 |
| 关联模块 | `server/nodes/prompts.py`, `server/intent/recognizer.py` |
| 关联 Spec | Spec-003（Intent路由）, Spec-004（会话管理） |

## 一、背景与现存问题

项目面向 OPC 创业者，提供从需求调研→商业方案→产品设计→营销推广的完整创业流水线。每条用户输入需要被路由到正确的处理节点，每个节点有自己的系统提示词和输出结构。如果不做显式节点拆分，所有逻辑压缩在一个 prompt 中，LLM 无法针对性优化，输出质量不可控。

## 二、最终实现方案

**9 个 Node 节点**，分为两类：

| Node | 名称 | 职责 | 需要联网搜索 |
|------|------|------|------------|
| node1 | 市场调研 | 行业数据收集、竞品分析、用户画像 | 是 |
| node1.5 | 品牌设计 | 人群精神萃取→品牌屋→三合一提示词包 | 否 |
| node2 | 商业方案设计 | 商业模式画布+品牌体系（命名/理念/故事） | 否 |
| node3 | 产品设计 | SKU规划、定价、最小启动方案 | 否 |
| node3.1 | 图片生成 | Logo/效果图渲染（调用 Seedream） | 否 |
| node4 | 营销方案设计 | 7步内容营销框架（5A漏斗+4媒介+3个月） | 否 |
| node5 | 营销文案 | 平台种草文案、短视频脚本、私域话术 | 否 |
| export | 文件导出 | 对话内容→HTML/PPT/PDF | 否 |
| fallback | 智能助手 | 非流水线需求兜底 | 否 |

**依赖链**（`DEPENDENCIES` 字典定义）：

```
node1 ────────────────┐
                      ├──→ node2 ──┬──→ node3
                      │            │
                      │            ├──→ node3.1
                      │            │
                      └──→ node4 ←─┘ (node1.5 为 soft 依赖)
                                │
                      node1.5 ──┘
```

- **hard 依赖**：缺失时**阻断**执行，提示用户先完成前置节点
- **soft 依赖**：缺失时静默跳过，不阻断但可能影响输出精度

| 目标节点 | hard 依赖 | soft 依赖 | 设计原因 |
|---------|----------|----------|---------|
| node1 | 无 | 无 | 调研是起点，无前置依赖 |
| node2 | node1 | 无 | 商业方案需要市场规模/竞品/用户数据 |
| node1.5 | node1, node2 | 无 | 品牌设计必须先有商业模式再做品牌 |
| node3 | node2 | 无 | 产品设计基于商业模式确定SKU方向 |
| node4 | node1, node2 | node1.5 | 营销需要市场数据+商业方案；品牌精神有更好，不强制 |
| node5 | 无 | 无 | 营销文案可独立触发，不强制前置 |

## 三、做出该选择的底层原因

**业务原因**：OPC 创业者的决策流程天然是线性的——先看市场机会（node1），再定商业模式（node2），再做产品（node3），最后做营销（node4/node5）。节点顺序匹配真实创业决策流，用户使用无认知负担。

**技术原因**：
1. 每个节点有独立的 system prompt（300-1000字），拆分后 prompt 更聚焦，LLM 输出质量更高
2. 节点间的依赖校验避免"跳跃式"使用——比如没做市场调研就直接要求生成营销方案

**对比过的备选方案**：
- 备选A：所有功能合并在一个巨型 prompt 中 → 弃用：LLM 无法在2000字 prompt 中兼顾所有场景，输出质量差
- 备选B：完全自由路由，不做依赖校验 → 弃用：用户跳过调研直接做方案时，输出内容空洞无数据支撑
- 备选C：强依赖链（node3 也依赖 node1.5）→ 弃用：增加不必要的阻塞，产品设计不需要品牌精神也能做

## 四、落地硬性约束

- 新增 Node 必须在 `NODES` 和 `NODE_SYSTEM_PROMPTS` 两处同步注册
- 新增 Node 若有前置依赖，必须在 `DEPENDENCIES` 和 `NODE_TRIGGERS` 中声明
- Node system prompt 必须包含：上游输入说明、输出结构、数据铁律
- 禁止在 Node prompt 中引用未在 DEPENDENCIES 中声明的前置节点数据

## 五、功能边界

### 覆盖场景
- 用户描述创业想法 → intent识别 → 路由到匹配Node → 生成结构化输出
- 追问场景 → `detect_followup_type` 判断 → 延续当前Node上下文
- 跨Node触发 → 依赖校验 → 阻断或放行

### 明确不覆盖
- 不处理多Node并行执行（单次请求仅一个Node）
- 不支持用户自定义Node（Node定义是开发层的配置）
- fallback 不执行任何流水线业务逻辑

## 六、不这么做的风险

1. 无Node拆分：单prompt输出质量不可控，调研和营销文案混在同一个回复中，用户无法获取结构化结果
2. 无依赖校验：用户跳过关键步骤直接请求方案时，LLM 编造数据填补空白，违反数据真实性铁律
