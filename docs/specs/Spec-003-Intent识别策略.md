## 一、背景与现存问题

用户输入从"调研网球服"到"刚才那个方案展开品牌定位"到纯闲聊，差异巨大。如果全部调 LLM 做意图识别，简单关键词匹配就能解决的场景会浪费 1-3 秒的 API 调用时间。如果全用关键词匹配，复杂意图无法正确路由。

## 二、最终实现方案

**四层漏斗式匹配**，优先级从高到低：

```
用户输入
    │
    ▼
第一层：关键词优先强匹配
    │  条件：同时命中"调研/行业/赛道"类 + 具体品类词
    │  动作：直接返回 node=1，不调 LLM
    │  命中则跳过后续层
    ▼
第二层：报告/总结路由
    │  条件：命中"生成报告/品牌手册/商业计划书"等
    │  动作：返回 node=summary 或对应的特定节点
    │  命中则跳过后续层
    ▼
第三层：LLM 意图识别
    │  条件：前两层均未命中
    │  动作：调 DeepSeek 执行 INTENT_PROMPT，JSON 返回 node
    │  LLM 返回 fallback → 进入第四层
    ▼
第四层：关键词兜底匹配
    │  条件：LLM 调用失败或返回 fallback
    │  动作：15组关键词映射表逐条匹配
    │  仍未命中 → 最终返回 node=fallback
```

**追问检测（独立于Node路由）**：

`detect_followup_type()` 独立判定输入是全新请求还是追问：
- 追问特征词（"刚才/这个/补充/展开"）+ 短句（<20字）+ 有历史上下文 → 判定为追问
- 报告类命令（含"市场调研报告/商业方案/品牌手册"等关键词）→ **强制视为新请求**，不走追问
- 追问情况下跳过 intent 识别，直接复用当前 `_session_state.last_node`

## 三、做出该选择的底层原因

**业务原因**：大部分用户输入可以用关键词覆盖——"调研XX行业"占了流量的70%。第一层keyword匹配让这70%的请求在毫秒级完成路由，不需要调LLM，节省API费用和延迟。

**技术原因**：
1. 业界"调研"+"品类词"的组合对Node1的匹配准确率接近100%，LLM 识别反而可能出错（LLM 可能把"调研"理解为"研究一下某个概念"）
2. LLM 调用作为第三层兜底，覆盖关键词无法处理的模糊输入
3. 第四层是最后的安全网——即使 LLM 超时/报错，仍有确定性规则可以工作

**对比过的备选方案**：
- 备选A：全部走LLM识别 → 弃用：高延迟+高费用，简单输入没必要
- 备选B：全部走关键词 → 弃用：复杂意图（如"我想做一个类似XX但针对YY人群的产品"）无法正确路由
- 备选C：LLM优先、关键词兜底 → 弃用：顺序反了——关键词更精确，应该优先

## 四、落地硬性约束

- 第一层品类词列表可扩展，但新增品类必须同时更新 `category_words`
- `REPORT_TRIGGERS` 列表变更需要同步更新到追问检测和 `recognize_intent`
- LLM 返回的 node 值必须是 `"1"|"2"|"3"|"3.1"|"4"|"5"|"fallback"` 之一
- 第四层兜底匹配前必须做黑名单校验（防止LLM误将调研需求路由到报告节点）

## 五、功能边界

### 覆盖场景
- 明确赛道+调研需求 → 第一层直接命中
- 模糊创业想法 → 第三层LLM分析
- 追问/微调 → 追问检测，不走意图识别
- 报告生成命令 → 第二层路由

### 明确不覆盖
- 不支持多意图并列（"帮我调研市场然后写个营销方案"→ 仅识别第一个意图）
- 不支持用户显式指定Node（无 `/node2` 类命令）
- Node 依赖校验不属于 intent 层职责（由 `validate_node_prerequisites` 独立处理）

## 六、不这么做的风险

1. 全部LLM路由：每次路由多花1-3秒，日请求量上来后API成本不可忽视
2. 全部关键词路由：复杂意图错误率高，用户输入"我想做一个像lululemon但更平价、主打Z世代女性的运动品牌"会被错误路由或兜底
3. LLM优先于关键词：调研类请求被LLM误判为fallback的比例实测约5%，浪费用户时间
