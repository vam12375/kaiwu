# Spec-006: API 密钥禁止硬编码

| 字段 | 内容 |
|------|------|
| Spec 类型 | 绝对不做 |
| 冻结状态 | 已冻结 |
| 创建日期 | 2026-06-30 |
| 关联模块 | `server/config.py` |
| 关联 Spec | 无 |

## 一、禁止的功能 / 技术方案

**禁止将任何 API 密钥、Token、密码以字符串字面量形式写入源代码文件。**

具体包括：
- 禁止 `API_KEY = "sk-xxxxx"` 形式的硬编码
- 禁止在注释中写密钥（"测试用 key: sk-xxx"）
- 禁止在 commit message 或 PR 描述中暴露密钥文本

## 二、否决的完整原因

**触发事件**：基线 Review（2026-06-30）发现 `config.py` 中硬编码了 3 把 API 密钥。当日已通过 P0-1 修复完成——移到 `.env` 文件。

**否决原因**：

1. **安全层面**：代码进入 Git 仓库后，密钥永久存在于 Git 历史中。即使后续 commit 删除密钥行，`git log -p` 依然可查阅历史版本中的密钥文本。Git 的不可变性意味着一次提交 = 永久泄露。

2. **运维层面**：密钥与代码绑定后，轮换密钥（如应对泄露、定期轮换）需要修改代码 + 重新部署，无法通过配置管理独立操作。

3. **架构层面**：开发/测试/生产环境需要使用不同的 API 密钥。硬编码使得每个环境需要一份不同的 `config.py`，产生配置分支，增加合并冲突概率。

**曾考虑的相关方案**：
- "加密存储密钥，代码中写解密逻辑" → 否决：解密密钥还是要存，没有解决根本问题，只是多了一层花哨包装
- "用私有仓库就不怕" → 否决：私有仓库也有权限误配、离职员工账号未回收等泄露路径

## 三、长期约束规则

- 所有密钥必须通过环境变量或 `.env` 文件注入
- `.env` 文件必须在 `.gitignore` 中排除
- `.env.example` 模板文件不含真实密钥，可安全提交到仓库
- 新增任何外部 API 集成时，密钥管理同样遵循此规则
- `config.py` 读取密钥使用 `os.getenv("KEY_NAME", "")`，默认值必须为空字符串（不做默认密钥）
- CI/CD 流程中密钥通过 secrets manager 注入，不依赖 `.env` 文件

## 四、变更该决策需要的评审条件

满足以下**全部条件**方可重新评审此 Spec：

- [ ] 有新的业界标准方案证明环境变量分离方式已不再是最佳实践
- [ ] 新方案提供了不低于当前方案的安全级别
- [ ] 迁移成本被评估为可接受（不要求整个代码库同时迁移）
- [ ] 项目架构 Review 会议通过

## 五、违反此 Spec 的后果

1. **密钥泄露**：代码分发或仓库公开后，API 密钥被第三方获取，产生非授权调用费用
2. **轮换困难**：密钥泄露后变更流程涉及代码改动+部署，响应时间从分钟级变成小时级
3. **审计失效**：无法追溯每个环境使用了哪把密钥，安全审计时无法提供密钥生命周期记录
